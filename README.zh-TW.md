# Cognitive Hub

**以記憶為核心的 AI 代理設計模式，專為多角色知識工作打造。**

外部化的活記憶（externalized living memory）——人類可讀、AI 可讀、可協作編輯。

> 🌐 [English version](README.md)

---

## 問題

知識工作者——醫師、研究者、管理者、律師、工程師——面臨的瓶頸不是「找不到資訊」，而是「同時扮演太多角色」。

你早上以研究者身份做了決定，下午切換到行政工作，三天後需要回到那個研究脈絡——但你已經記不清當初考慮過哪些選項、排除了什麼方案、為什麼做出那個選擇。這不是記憶力問題，而是人類認知的根本限制：我們能同時維持的工作脈絡有限，切換角色時，前一個角色的決策記憶會流失。

目前 AI 的主流方向是「幫你做事」——自動寫信、管行事曆、跑自動化流程。這解決的是「手不夠用」。但對於多角色知識工作者，更核心的困境是「腦不夠用」：

- 跨角色切換時，前一個角色的決策脈絡遺失
- 過去的經驗和教訓沒有被系統性累積，同樣的錯誤反覆發生
- 團隊成員之間缺乏共享的決策記憶，溝通成本高
- AI 每次新對話都從零開始，你需要反覆解釋背景

**Cognitive Hub 解決的是「腦不夠用」——讓 AI 成為你的認知夥伴，而不只是任務執行者。**

## 架構

Cognitive Hub 是一個**設計模式（design pattern），**它可以用任何 LLM + 結構化知識平台 + 橋接層的組合來實現。參考實作使用 Claude + Notion + MCP，但架構本身不綁定任何特定工具。

### 三層記憶架構

![三層記憶架構](docs/architecture.svg)

```
Layer 1 — 用戶偏好（每次對話自動載入）
├── 核心身分、溝通規則、路由指令
├── 最小佔用空間——每次對話開始時載入
└── 告訴 AI：「去讀我的 Hub 取得脈絡」

Layer 2 — 知識庫（按需載入）
├── Rules      — 決策邏輯與原則
├── Projects   — 進行中的工作與狀態
├── References — 事實、聯絡人、任務來源
├── SOP        — 重複性工作的標準流程
└── Prompts    — 可重用的 prompt 模板

Layer 3 — 版本控制備份（選用）
├── Hub 的純文字 (.md) 鏡像
├── 不依賴特定平台的保險
└── Git 歷史，提供變更追蹤
```

**運作方式：** AI 自動載入 Layer 1 → 讀到指令後透過橋接層讀取 Layer 2 中與當前任務相關的頁面 → 帶著完整脈絡回應。Layer 3 是安全網，不在日常工作流中。

### 核心設計決策

**記憶優先。** 多數 AI agent 框架追求自主任務執行——AI 替你做事。Cognitive Hub 反轉這個優先序：著重知識先行，行動建立在共享的結構化脈絡之上。AI 不只是做，而是*理解*。

**顯式規則。** 規則、決策、工作流程寫在結構化頁面裡——不埋在對話歷史中，也不靠 AI 推測。你可以閱讀、編輯、分享它們。修改一條規則，AI 的行為立即改變。

**人機共同維護。** Hub 不是上傳一次的靜態文件。人和 AI 都能讀寫。AI 在新決策後可能建議更新規則，人審核確認。隨時間推移，Hub 成長為你工作方式的活記錄。

**與工具無關。** 分層記憶、顯式規則、異步接力、收割思維——這些設計模式可在任何平台上實現。從 Notion 換到 Obsidian，從 Claude 換到其他 LLM，累積的知識都能跟著你走。

## 核心能力

### 跨角色決策記憶

當你從研究者切換到管理者、再切換到臨床工作，AI 已經掌握你在每個角色中的決策內容、理由和未完事項——無需重新說明。

### 規則驅動行為

不再期望 AI「猜到」你的偏好——而是寫下明確的規則：溝通風格、決策框架、研究協定、行政程序。AI 讀取並遵循。

### 異步接力工作流程（Async Relay Workflow）

團隊的協作模式：主管與 AI 構想 → 建立共用知識頁面（目標、方法、待辦、經驗紀錄）→ 交接給團隊成員各自接力。不需要開會就能保持脈絡一致。

### 收割思維（Harvest Mindset）

當建立新工具或 pipeline 時，AI 主動評估發表潛力。有可發表的角度：*「這個可以收割——建議多做這一步。」* 純行政工具：*「停在 MVP 即可。」* 防止有價值的工作被忽略。

### 整合式個人生產力

信件分流、行事曆管理、語音筆記處理、任務路由——全都根據 Hub 中你的偏好、優先序和現有承諾來執行。

## 應用情境

參考實作由一位教學醫院醫師開發。他同時管理六個角色：臨床判讀、學術研究、科務行政、團隊協調、教學和技術開發。每個角色持續產生碎片化的決策、規則和待辦事項，散落在不同工具和時間點。

Cognitive Hub 不為每個角色建立獨立系統，而是提供單一知識架構——所有角色共享同一個記憶層。AI 可以隨時調取任何角色的相關脈絡：研究決策影響行政簽呈、臨床觀察觸發發表建議、團隊交辦頁面記錄的教訓避免重複犯錯。

這個情境是示範，不是限制。此架構適用於任何同時承擔多重職責的知識工作者。

## 與現有方案的比較

### vs. 個人知識管理（Personal Knowledge Management, PKM）

Zettelkasten、PARA、Building a Second Brain 等方法專注於「人如何組織知識」，不涵蓋 AI 讀寫能力。Cognitive Hub 加入了雙向性：人寫給 AI 讀，AI 寫給人審。知識庫同時服務兩種讀者。

### vs. 行動優先的 AI Agent

主流 agent 框架追求自主任務執行——AI 執行 shell 指令、發訊息、管理檔案。記憶是副產品（對話歷史、skill 目錄）。Cognitive Hub 反轉這個架構：結構化記憶是基礎，行動建立在記憶之上。規則、決策歷史、團隊分工、研究策略都是顯式且可編輯的——不是隱含在聊天記錄裡。

| 維度 | 行動優先 Agent | Cognitive Hub |
|------|---------------|---------------|
| 核心定位 | 自主任務執行 | 知識增強的決策支援 |
| 記憶模型 | 隱式（對話歷史） | 顯式（分層、人機共同維護） |
| 規則系統 | Skills（任務導向） | Rules / SOP / Protocols（決策導向） |
| 團隊協作 | 單人為主 | 多人共享知識庫 |
| 適用對象 | 開發者 / 進階使用者 | 知識工作者（不需 command line） |
| 建置門檻 | Runtime + Docker + messaging 平台 | 知識平台帳號 + LLM 帳號 + 橋接層 |

### vs. LLM 內建記憶

主流 LLM 已提供記憶功能，但這些記憶：由 AI 自動提取，使用者控制有限；對團隊成員不可見；沒有分層架構；且鎖定在特定平台。Cognitive Hub 的外部化記憶由使用者控制、有結構、可分享，且可跨 LLM 平台攜帶。

### vs. Custom GPTs / Claude Projects / Gems

這類工具允許上傳靜態文件作為 context。Cognitive Hub 的記憶是動態的——每次對話都可能更新規則、新增專案、修改 SOP。透過橋接層（MCP、API），AI 不只讀取文件，還能主動寫入知識庫。

## 範本

`templates/` 目錄包含去識別化的範例模板：

- **`rules-template.md`** — Rules 頁面範本，展示如何結構化決策邏輯供 AI 讀取
- **`project-template.md`** — 異步接力工作流程的專案頁面範本（目標、方法、角色分工、待辦、經驗紀錄）
- **`sop-template.md`** — 標準作業流程範本（含 AI 指令段落）

這些是起點。請根據你的角色、工具和工作方式自由調整。

## 快速開始

你需要兩樣東西：

1. **一個可存取外部知識的 LLM** — Claude + Notion MCP、ChatGPT + plugins，或任何可以讀寫外部知識平台的 LLM
2. **一個結構化知識平台** — Notion、Obsidian、Confluence，或任何支援結構化頁面且 AI 可存取的工具

設定約需一小時：

1. 建立 Hub 主頁，下設子區塊：Rules、Projects、References、SOP
2. 寫 5–10 條核心規則（溝通風格、決策原則、AI 不該做的事）
3. 加入基本 References（你是誰、你的角色、你的工具和偏好）
4. 設定你的 LLM 在對話開始時讀取 Hub
5. 測試：*「你知道我目前有哪些專案嗎？」*

詳細設定指南：

## 設計原則

以下原則構成了架構的理論基礎，記錄於此以供複製：

1. **筆記的讀者已經改變。** 在 AI 時代，筆記同時服務人和 AI。知識系統必須讓兩者都能讀寫。
2. **為 AI 儲存，為人類呈現。** 記錄階段優先保存資訊密度和決策邏輯。人需要閱讀時，由 AI 動態生成適合的版本。
3. **認知分身：知識與運算引擎分離。** LLM 是可替換的運算引擎。你的決策邏輯、管理原則、研究框架必須獨立於任何特定模型。
4. **人的價值在定義問題，不在操作 AI。** 當 AI 降低執行成本，操作技能會快速商品化。真正不可替代的是：定義有價值的問題、設定判定標準、驗證結果。
5. **資料層比工具層長壽。** 工具的壽命短於知識的壽命。知識庫的設計不應依附於單一工具。

## 限制

- **單一團隊驗證。** 此架構由一個機構的一個團隊開發和測試。需要更廣泛的跨領域、跨規模驗證。
- **尚無正式使用者研究。** 使用成果基於作者經驗，非控制實驗。
- **橋接層依賴平台支援。** 雖然設計模式是工具無關的，但橋接層（如 MCP）取決於特定平台支援。並非所有 LLM + 知識平台組合都有成熟的橋接方案。
- **維護成本。** Hub 需要主動維護。過時的 Hub 比沒有 Hub 可能更糟——AI 會帶著自信使用過期的脈絡。
- **隱私考量。** 將 LLM 連接到你的知識庫意味著 LLM 服務商會處理這些內容。使用者需評估自身的接受程度和機構政策。
- **不替代機構級系統。** Cognitive Hub 是個人/團隊的生產力架構，不是企業知識管理系統。
